● Based on all architecture docs, here are the general components every Skill 
  depends on (regardless of which specific ports it uses):

  ---
  1. Domain Contracts (src/domain/)

  All skill inputs and outputs are Pydantic v2 models from this module. Skills
  never use raw vendor JSON — domain types are the universal contract boundary
  (ADR-004).

  2. Ports layer (src/agents/skills/<skill>/)

  Skills declare their port dependencies and receive them via constructor
  injection at compose time. They never import adapters directly — only port
  protocols.

  3. Persistence (src/persistence/)

  LangGraph checkpoints skill-local state through src/persistence/ — required
  for HITL pause/resume and fault recovery. Skills never call storage SDKs
  directly.

  4. Observability / LangSmith

  Every skill run must tag its LangSmith trace with case_id, workflow_id, and
  step_id before any LLM call. Structured JSON logs with correlation ID are
  emitted on state transitions.

  5. LangGraph runtime

  Skills are LangGraph subgraphs — the framework itself is a hard runtime
  dependency providing graph execution, interrupt() for In-Skill HITL, Send for
  fan-out, and checkpointing hooks.

  ---
  In summary: Domain Contracts + Ports (injected) + Persistence + Observability 
  + LangGraph are the four architectural components every Skill depends on,
  independent of its specific business function.

❯ What is your opinion: do we need LangGraph Tools as a separate components?    

● Yes, I think Tools should be a distinct component. Here's why:

  The core reason: Skills contain LLM reasoning nodes. The LLM doesn't call a
  port method directly — it emits a structured function-call decision. Something
   has to translate that into a port invocation. That's exactly what a LangChain
   Tool does: it exposes a name, description, and args_schema that the LLM
  reasons about, then dispatches to the port.

  Without a Tool layer, a Skill's internal nodes must call ports imperatively —
  making the flow fully deterministic, with no LLM-driven selection between
  capabilities. That undermines the point of having LLM reasoning inside a
  skill.

  Concrete benefits of separating Tools:

  - LLM-addressable — the LLM can dynamically choose which tool (port) to invoke
   based on context
  - Testability — Tools are a natural seam; you can swap them in tests without
  touching skill graph logic
  - Observability — LangSmith traces tool calls as first-class events, giving
  per-invocation latency, inputs, outputs
  - Reusability — a Tool wrapping LlmPort or RagPort can be shared across
  multiple Skills without duplication

  What this means for the architecture doc:

  The chain should be explicitly documented as:

  Skill → Tool (LangChain Tool wrapping a Port) → Port → Adapter → External
  system

  And Tools should appear as a named component in agents.md, not just as an
  arrow label in the mermaid diagram.

● Fakes are not a runtime dependency of a Skill — they're a testing-time 
  substitute for ports/adapters. So they don't belong in the general runtime
  dependency list.

  However, quality-and-delivery.md makes a strong rule: every new port must ship
   with a matching fake in tests/fakes/. So fakes are a mandatory delivery
  dependency — a Skill isn't considered complete without them.

  The distinction:

  ┌───────────┬──────────────────────────────────────────┐
  │  Context  │                Dependency                │
  ├───────────┼──────────────────────────────────────────┤
  │ Runtime   │ Tools → Ports → Adapters                 │
  ├───────────┼──────────────────────────────────────────┤
  │ Test / CI │ Tools → Ports → Fakes (replace Adapters) │
  └───────────┴──────────────────────────────────────────┘

  If you want the architecture doc to reflect this, it's worth adding a note in
  agents.md that every Tool/Port a Skill declares requires a corresponding fake
  before the skill can merge — mirroring the rule already in
  quality-and-delivery.md.
